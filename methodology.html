WebGPU Compute Throughput: A Deterministic Model for Browser-Based Hardware Validation

Dr. Aris Thorne, PhD • Department of Systems Architecture • Feb 2026
Abstract: This paper outlines the methodology used by GearVerify to measure hardware capabilities within a sandboxed browser environment. By utilizing the `timestamp-query` extension in WebGPU, we decouple JavaScript event loop latency from the actual GPU command execution time, allowing for sub-millisecond precision in measuring Floating Point Operations (FLOPS) and memory bandwidth without native OS hooks.

1. The Latency Problem

Traditional browser benchmarks rely on `performance.now()`, which measures time on the CPU timeline. This introduces jitter caused by the browser's main thread layout and garbage collection. To solve this, GearVerify measures the Total System Latency ($L_{total}$) as a sum of discrete GPU timeline events plus driver overhead.

$$ L_{total} = \sum_{i=0}^{n} (T_{dispatch} + T_{execution} + T_{readback}) + \epsilon_{driver} $$

Where:

  • $T_{dispatch}$: The time to encode commands into the command buffer.
  • $T_{execution}$: The raw silicon execution time measured by GPU timestamp queries.
  • $T_{readback}$: The cost of mapping the `GPUBuffer` back to CPU-readable memory.
  • $\epsilon_{driver}$: The non-deterministic noise introduced by the WDDM (Windows) or Metal (macOS) kernel scheduler.

2. Measuring GPU Throughput (TFLOPS)

To calculate the theoretical maximum throughput, we saturate the ALUs with a fused multiply-add (FMA) workload. The compute shader calculates a $1024 \times 1024$ matrix multiplication. The throughput $P$ is derived as:

$$ P_{TFLOPS} = \frac{2 \times M \times N \times K}{T_{execution} \times 10^{12}} $$

This formula accounts for the accumulator (multiply + add = 2 operations). By isolating $T_{execution}$ using hardware timestamps, we eliminate the "Browser Tax" typically associated with WebGL.

3. Zero-Server Privacy Protocol

GearVerify executes this math entirely within a local V8 Isolate. The results ($P_{TFLOPS}$) are rendered to the DOM but never transmitted via POST/GET requests. This ensures cryptographic certainty that your hardware fingerprint remains local.
